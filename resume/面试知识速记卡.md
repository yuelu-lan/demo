## 一、React 原理(四点)

### 1. Fiber 架构

- **是什么**:React 16 把递归同步渲染(Stack Reconciler)换成基于链表的可中断渲染(Fiber Reconciler)。
- **为什么需要**:React 15 递归调用栈一旦开始停不下来,大组件树阻塞主线程 → 卡顿。Fiber 把组件树拍成链表,渲染变成可分片、可中断、可恢复的任务,每帧留时间给浏览器响应交互。
- **Fiber 节点字段**:`type`(组件)、`stateNode`(真实 DOM)、`child/sibling/return`(树形关系)、`memoizedProps/pendingProps`、`lanes/expirationTime`(优先级)。
- **能中断恢复的原因**:Fiber 是链表,记录"渲染到哪了",中断后从断点继续;递归调用栈中断即丢上下文。

### 2. Diff 策略(三个假设,降 O(n³) → O(n))

- **跨层不 diff**:节点跨层移动直接销毁重建,不智能搬运。
- **同类型才 diff**:不同类型(`<div>`→`<span>`、`CompA`→`CompB`)直接销毁重建。
- **list 靠 key**:同层兄弟用 `key` 标识身份,相同复用、变了重建。
- **为什么不能用 index 当 key**:列表增删时 index 错位,React 复用错节点,导致 state 串、输入框内容错乱、动画抖动。要求 key 稳定且唯一。

### 3. 批量更新(Batching)

- **合成事件 / 生命周期里** → batch(异步合并),连续 `setState({a:1})`、`setState({a:2})` 最终 `a=2`。
- **setTimeout / Promise / 原生事件里** → React 15/16/17 **同步**(每次触发 render);React 18 **自动 batch**(`createRoot` 引入 Automatic Batching)。
- **实现差异**:React 15 用 `transaction`(事务);React 16+ 用 Fiber + `lane` 优先级调度。
- **外部表现**:15 vs 16 在合成事件里一样,真区别在非合成事件场景(15/16 同步 vs 18 自动批量)。

### 4. Hooks 机制

- **状态存哪**:存在对应 Fiber 节点的 `memoizedState` 上,是一条**单向链表**(每个 hook 一个节点,`next` 串起)。
- **为什么不能放 if/循环里**:函数组件每次 render 重新执行,React 按**调用顺序**从链表取对应节点 —— 顺序错了就错位。顶层调用是硬性要求。
- **闭包陷阱**:`useEffect` 闭包捕获首次 render 的变量,deps 空时定时器里的 `count` 永远是首次值。解法:依赖数组补全 / `useRef` / 函数式更新 `setCount(c => c+1)`。

### (补充)Concurrent Mode / 并发渲染

- React 18 默认开启,基于 Fiber 可中断能力。
- 三个词:**时间分片**(大渲染拆小片不阻塞交互)、**优先级调度**(用户输入优先于数据 fetch,可打断低优先级渲染)、`useDeferredValue`/`useTransition`/`Suspense` 基于此。

## 二、`new` 与原型链

### `new` 做的四件事

1. 创建一个全新的空对象 `obj = {}`。
2. 把 `obj.__proto__` 指向构造函数的 `prototype`(`obj.__proto__ = Person.prototype`),建立原型链。
3. 把构造函数 `this` 绑到 `obj` 并执行(`Person.call(obj, '阴文喻')`),给实例属性赋值。
4. 判断返回:构造函数**显式 return 对象**则返回该对象;否则返回 `obj`。(`return` 原始值/`null` 会被忽略。)

### 手写实现

```js
function myNew(constructor, ...args) {
  const obj = Object.create(constructor.prototype);  // 1+2: 新建空对象并挂原型链
  const result = constructor.apply(obj, args);       // 3: 绑定 this 并执行
  return (result instanceof Object) ? result : obj; // 4: 对象则返回,否则返回 obj
}
```

- `Object.create(proto)` = 创建以 `proto` 为原型的新对象,等价于完成了第 1、2 件。
- `result instanceof Object`:`null` 和原始值(数字/字符串/布尔)都为 `false`,正好走 `obj`;对象为 `true` 走 `result`。
- `null instanceof Object` 是 `false`,所以 `return null` 也正确处理成返回 `obj`。

### 必记的三条链路

```
实例.__proto__              === 构造函数.prototype
构造函数.prototype.__proto__ === Object.prototype
Object.prototype.__proto__   === null   ← 链终止
```

对称的另一条(构造函数自身也是对象,由 Function 构造):

```
构造函数.__proto__ === Function.prototype
Function.prototype.__proto__ === Object.prototype
```

## 三、事件循环(Event Loop)

### 一次循环的结构

> ① 执行一个宏任务 → ② 清空**全部**微任务 → ③ 渲染(UI repaint,如有必要) → ④ 取下一个宏任务

- **微任务是"清空全部",宏任务是"只取一个"** —— 这是和"微任务→宏任务"含糊说法的关键区别。
- **当前同步脚本本身就是第一个宏任务**。

### 经典题

```js
console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
console.log('4');
```

输出:`1 4 3 2`
- 同步代码(第一个宏任务)打印 1、4。
- `setTimeout` 回调入宏任务队列,`Promise.then` 入微任务队列。
- 当前宏任务结束 → 清空微任务打印 3 → 取下一个宏任务(setTimeout)打印 2。

### 为什么 setTimeout(0) 必在 Promise.then 之后

`setTimeout` 回调属下一轮宏任务,要等当前轮微任务清空完才轮到;`Promise.then` 在当前轮"清空微任务"阶段就执行。一个在本轮,一个在下一轮,所以 then 先于 setTimeout —— 不是笼统的"微任务优先级高"。

### 分类

- **微任务**:`Promise.then/catch/finally`、`queueMicrotask`、`MutationObserver`、`process.nextTick`(Node)。
- **宏任务**:`setTimeout`、`setInterval`、`setImmediate`(Node)、I/O、UI 渲染、`postMessage`。

### async/await

- `await x` = 把 `await` 后面剩下的函数体包成 `.then` 回调,挂到 `x` 的 Promise 上。**不阻塞主线程**,只是挂起当前 async 函数。
- `await b()` 会**立即执行 b 的同步部分**;b 返回 resolved Promise 后,`await` 之后的代码作为**微任务**推入队列。

```js
async function a() {
  console.log('a start');
  await b();
  console.log('a end');
}
async function b() { console.log('b'); }
console.log('1'); a(); console.log('2');
```

输出:`1 a start b 2 a end`
- 打印 1 → 进 a 打印 a start → `await b()` 立即执行 b 打印 b,b 返回 resolved Promise,`a end` 作为微任务挂起,a 函数返回让出主线程 → 打印 2 → 清空微任务打印 a end。

**核心认知**:`await` 后的代码 ≈ `.then` 回调,是微任务,不阻塞主线程。

## 四、Webpack 插件基础

### Loader vs Plugin

| | Loader | Plugin |
|---|---|---|
| 本质 | 文件内容转换器(翻译器) | 构建流程钩子(扩展器) |
| 作用对象 | 单个文件的内容 | 整个构建流程和产物 |
| 执行时机 | 模块解析时,逐个文件转换 | 贯穿构建生命周期 |
| 能做什么 | 非 JS → JS、TS→JS、less→css | 打包优化、资源管理、注入环境变量、清 dist、生成 HTML、压缩 |
| 接口形式 | 一个函数(输入源码,输出源码) | 一个类,带 `apply` 方法 |

一句话:**Loader 转文件,Plugin 钩流程。**

### Loader 长什么样

```js
module.exports = function (source) {
  return source.replace('old', 'new');  // 输入源码字符串,返回转换后源码
};
```

- 改 AST 的是 **Babel 插件**,它在 Webpack 里通过 **babel-loader**(Loader)调用,不是 webpack plugin。
- 链路:webpack → babel-loader → @babel/core → 你的 babel 插件(改 AST)。

### Plugin 长什么样

```js
class MyPlugin {
  apply(compiler) {                                    // 必须有 apply
    compiler.hooks.emit.tapAsync('MyPlugin', (compilation, cb) => {
      // compilation.assets 是打包产物,可增删改
      cb();
    });
  }
}
module.exports = MyPlugin;
```

必记三点:
1. **Plugin 是 class,必须有 `apply` 方法** —— Webpack 启动时调 `plugin.apply(compiler)` 注入。
2. **事件机制叫 Tapable(hooks)** —— `.tap` 注册同步,`.tapAsync`/`.tapPromise` 注册异步。
3. **`compiler` vs `compilation`**:`compiler` 整个生命周期全局唯一;`compilation` 每次构建新建一次,含模块资源、chunk、`assets`。

常见钩子名:`entry-option`、`compile`、`emit`(输出前,最常用,能改 assets)、`after-emit`、`done`。

### 构建流程(留个印象即可)

> 读取配置 → 初始化 compiler → 从 entry 递归解析依赖(建依赖图)→ loader 转换每个模块 → 输出 chunk/assets → 写入 dist。

核心词:**入口、依赖图(dependency graph)、loader 转换、输出**。被问到说出"依赖图"就能过。
