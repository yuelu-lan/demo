## 第 1 题 · react15+antd2 → umi.js 架构升级的驱动力与方案选择

**【提问】**
这次升级的业务驱动力到底是什么?是技术债倒逼(antd2 维护困难、react15 无法用新生态),还是有具体业务诉求(新需求做不动、性能卡顿、安全合规)?在决定升级到 umi(bigfish)之前,评估过哪些备选方案(原地升 react16+antd3、迁移 qiankun 纯微前端、换 Vue),为什么选 umi?

**【我的回答】**
- 原架构太老,react15 无法使用 hooks、antd2 组件已停止维护且数量太少。
- 选 umi 是因为它是蚂蚁内部中后台标准框架。

**【参考答法 + 点评】**
答到点子但偏薄。
- 好的部分:技术债驱动说清了;选 umi 的真实理由(集团技术栈对齐、复用踩坑经验)很实在。
- 被追问的薄弱点:
  1. "组件数量太少"站不住 —— antd2→3 组件数差异不大,真痛点是 antd2 的 API 设计老旧(Form 的 create 包裹、getFieldDecorator)、bug 不再修、安全补丁停了。会被问"antd3 组件也不多,为何不升 antd3?"
  2. 没提业务侧驱动力。纯技术升级业务方视为零产出,升级 70 页意味着需求冻结。真实答案通常是"新需求依赖 hooks/新组件,老架构做不动"或"安全扫描 antd2 有 CVE 必须修"。
  3. 备选方案没说清。排除"原地升 react16+antd3"的理由:react15→16 生命周期变化 + antd2→3 breaking change 成本本身不低,而 umi 额外白送路由/构建/插件,投入产出比更高。
- **参考答法**:驱动来自两头,技术上 react15 不支持 hooks、antd2 停止维护、新需求里的 ProTable/ProForm 用不了;业务上新需求依赖的运营组件库只支持 react16+,老架构做不动,业务方愿投入。选 umi 而非原地升 react16+antd3,是因为升级成本本身不低,而 umi 作为蚂蚁中后台标准框架额外白送约定式路由、构建优化、插件机制,且团队后续项目都基于它,统一技术栈能复用踩坑经验。备选 qiankun 纯微前端评估过,但本项目单体应用、不涉及多团队拆分,引入是过度设计。

## 第 3 题 · babel 插件解决 ProComponent

**【提问】**
1. ProComponent 为什么用不了(react 版本/antd 版本/依赖冲突)? 2. babel 插件做了什么?babel 是编译期语法转换工具,而依赖版本是运行期问题,babel 怎么解决?

**【我的回答】**
1. ProComponent 依赖 antd 最新版,项目 antd 是 2.x,虽然装了 antd5 但别名安装,ProComponent 源码中指向的 antd 版本还是 2.x,使用时报错。
2. babel 插件在编译时修改 ProComponent 组件源码,将 import antd 的路径改为别名安装的 antd5。

**【参考答法 + 点评】**
答对了关键,表达要更精准。
- 正确的内核:根因是 ProComponent 源码写死 `import from 'antd'`,node 解析到根目录 antd2;babel 插件在编译期把 `import 'antd'` 改写成别名包路径 —— 思路正确,是处理"第三方包写死 import 路径"的常见手法。
- 要能接住的点:
  1. 为什么不用 webpack.resolve.alias 把 antd 整体指向 antd5?因为项目里还有 antd2 老页面也 `import 'antd'`,全局改会全挂。所以只能只改 ProComponent 内部的 import —— babel 插件比 webpack alias 更精准(能限定到某个 node_modules 包)。这个对比一定要讲。
  2. 怎么限定只改 ProComponent?babel 插件通过 include 过滤 node_modules 路径或在 visitor 里判断文件路径来源。
  3. 改的是哪个 AST 节点?ImportDeclaration,把 source.value === 'antd' 改成别名包。能说到 ImportDeclaration + source.value 这个 AST 级别,面试官就知道真写过。
- **参考答法**:ProComponent 依赖 antd4+,项目主体 antd2。虽然 npm alias 装了 antd5,但 ProComponent 源码写死 `import from 'antd'`,运行时被解析到根目录 antd2,API 对不上报错。没法用 webpack alias 把 antd 整体指向 antd5,因为还有 antd2 老页面也 import antd,全局改会全挂。所以写 babel 插件,在 ImportDeclaration 节点判断 source.value === 'antd',只对 ProComponent 包内(include 过滤 node_modules 路径)的源码改写,把 import 'antd' 替换成别名包路径。ProComponent 运行时拿到 antd5,老页面不受影响。

## 第 5 题 · React setState 批量更新(15 vs 16)

**【提问】**
React 的 setState 在 React 15 和 16+ 里行为有没有区别?同一合成事件里连续 setState({a:1})、setState({a:2}),同步还是异步?最终 state 是 1 还是 2?15 和 16 有无变化?

**【我的回答】**
react 源码太久没看,只记得 react16 会合并 setState。

**【参考答法 + 点评】**
诚实,但这里暴露了"源码级理解"的过度承诺。
- **参考答案**:同一合成事件里两次都是异步(不立即更新),最终 a=2(后覆盖前),这叫 batching。React 15 和 16+ 在合成事件里都会 batch,外部表现一样;区别在内部实现:15 基于 transaction(事务),16+ 基于 Fiber + expirationTime/lane 调度。真正让 batch 失效的场景是 setTimeout、Promise.then、原生事件监听 —— 15/16/17 里这些同步触发,React 18 Automatic Batching 后也 batch。这是 15 vs 18 的区别,不是 15 vs 16。
- **关键提醒**:简历"具备 React 源码层面理解"会被直接上 Fiber、Diff、Lane、Hooks 闭包等题。仅"会用 + 知道 setState 合并"撑不住。建议降级表述。

## 第 7 题 · 事件循环(宏/微任务)

**【提问】**
描述浏览器 Event Loop。console.log('1');setTimeout(()=>console.log('2'),0);Promise.resolve().then(()=>console.log('3'));console.log('4'); 打印顺序?宏任务/微任务区别?为什么 setTimeout(0) 必在 Promise.then 之后?

**【我的回答】**
1 4 3 2。因为事件循环机制是:主线程 → 微任务 → 宏任务 → 主线程。

**【参考答法 + 点评】**
顺序对,解释太薄。
- 完整流程:一次循环 = ① 执行一个宏任务 → ② 清空全部微任务 → ③ 渲染 → ④ 取下一个宏任务。
- 关键:微任务"清空全部",宏任务"只取一个",不是对等关系。当前脚本本身就是第一个宏任务。
- 为什么 setTimeout(0) 在 then 后:不是笼统"微任务优先级高"。setTimeout 回调属下一轮宏任务,要等当前轮微任务清空完才轮到;Promise.then 在当前轮"清空微任务"阶段就执行。一个本轮一个下一轮,所以 then 先。
- 分类:微任务 = Promise.then/catch/finally、queueMicrotask、MutationObserver、process.nextTick;宏任务 = setTimeout、setInterval、setImmediate、I/O、UI 渲染、postMessage。
- **核心记忆**:"清空全部微任务 vs 只取一个宏任务"是这道题的灵魂。

## 第 8 题 · 事件循环(async/await)

**【提问】**
async function a(){console.log('a start');await b();console.log('a end');} async function b(){console.log('b');} console.log('1');a();console.log('2'); 打印顺序?await 到底做了什么?

**【我的回答】**
1 / a start / 2 / b / a end。await 将 b 加入微任务,并阻塞 a 后面的代码。

**【参考答法 + 点评】**
顺序错,对 await 有关键偏差。正确顺序:1 / a start / b / 2 / a end。
- 错误1:b() 不是被 await "加入微任务"。b() 是普通函数调用,await b() 会立即执行 b 的同步部分(打印 b),b 同步跑完返回 resolved Promise。
- 错误2:await 不"同步阻塞"。它做的是:暂停 a 函数,把 await 之后的代码(a end)挂起,等 b() 的 Promise resolve 后,把 a end 作为微任务推入队列。a end ≈ .then 回调,是微任务,不阻塞主线程。
- 逐行:打印 1 → 进 a 打印 a start → await b() 立即执行 b 打印 b,b 返回 resolved Promise,a end 作为微任务挂起,a 函数返回让出主线程 → 打印 2 → 清空微任务打印 a end。
- **核心认知**:await x = 把 await 后剩下的函数体包成 .then 回调挂到 x 的 Promise 上。不阻塞主线程,只挂起当前 async 函数。

## 第 9 题 · 原型链与 new

**【提问】**
1. p.__proto__ === Person.prototype 值?为什么 new 能让实例 __proto__ 指向构造函数 prototype? 2. Person.prototype.__proto__ 指向什么?链终止于哪? 3. new 做了哪几件事?手写 myNew。

**【我的回答】**
1. true。2. Object.prototype,原型链最终归于 null。3. 不知道。

**【参考答法 + 点评】**
前两问对,第 3 问"不知道"是硬伤(简历"深入理解原型链"标配题)。
- **new 四件事**:① 创建空对象 obj;② obj.__proto__ 指向构造函数 prototype;③ 把 this 绑到 obj 执行构造函数(Person.call(obj, args));④ 判断返回:显式 return 对象则返回该对象,否则返回 obj(return 原始值/null 被忽略)。
- **手写**:
  ```js
  function myNew(constructor, ...args) {
    const obj = Object.create(constructor.prototype);
    const result = constructor.apply(obj, args);
    return (result instanceof Object) ? result : obj;
  }
  ```
- **必记链路**:实例.__proto__ === 构造函数.prototype;构造函数.prototype.__proto__ === Object.prototype;Object.prototype.__proto__ === null。
- **面试建议**:简历"深入理解原型链"建议改"熟悉原型链机制,理解 new、__proto__、prototype 的关系与继承实现"。用户选择不改(说一起学过只是忘了),回去背熟即可。

## 第 10 题 · Loader vs Plugin

**【提问】**
1. Loader 在哪个阶段执行?处理什么?为什么叫"翻译器"? 2. Plugin 和 Loader 本质区别? 3. 写一个最简单的 Plugin 要实现什么方法?Webpack 事件机制叫什么?

**【我的回答】**
webpack 只能处理 js,loader 让它能处理其他文件。plugin 可以处理 js 的 ast 树。

**【参考答法 + 点评】**
抓到一个对的点,两个明显错误。
- 对的部分:webpack 原生只懂 JS/JSON,Loader 把 CSS/图片/TS/less 转成 JS 模块,"翻译器"比喻正确。
- 错误1:"plugin 处理 js ast 树" —— 错。处理 AST 是 Loader(配合 babel)和 Babel 插件的活,不是 Plugin。Plugin 操作构建流程和产物,不碰 AST。这是把 babel 插件和 webpack plugin 搞混。
- 错误2:没回答本质区别。
- **核心区别**:Loader 是文件内容转换器(翻译器),作用于单个文件内容,模块解析时逐个转换;Plugin 是构建流程钩子(扩展器),作用于整个构建流程和产物,贯穿生命周期。一句话:Loader 转文件,Plugin 钩流程。
- **Plugin 形态**:class 带 apply 方法;事件机制叫 Tapable(hooks),.tap 同步/.tapAsync 异步;compiler 全局唯一 vs compilation 每次构建新建(含 assets)。常见钩子:emit(最常用,能改 assets)、done。
- **简历调整**:`具备插件开发能力` → `具备 Babel 插件开发能力`(已改)。

## 第 14 题 · Vue 2 的 data 为什么是函数

**【提问】**
Vue 2 的 data 为什么必须是一个返回对象的函数(data() { return {...} }),不能直接写成对象(data: {...})?写成对象会怎样?

**【我的回答】**
记不清了,好像是因为对象在 js 中存的是引用地址,修改对象后地址不变,vue 会认为对象没变,就不会更新?

**【参考答法 + 点评】**
理解偏了,但触及"引用"这个关键词。
- **正确答案**:不是为了"让 Vue 检测变化"(响应式检测靠 Object.defineProperty 的 getter/setter,和是不是函数无关)。真正原因是组件复用时每个实例要有独立 data,不能共享。写成对象,data 是对象(引用类型),所有实例共享同一对象引用,组件 A 改 count,B/C/D 全跟着变。写成函数返回对象,每次创建实例调用 data() 返回全新对象,每个实例拿到独立 data,互不影响。
- 一句话:组件是可复用的,data 必须是函数,才能保证每个实例拿到独立数据副本,不互相污染。
- 之前猜测"修改对象地址不变所以不更新"混淆了响应式检测机制,问题是数据隔离,不是变化检测。
- **结论**:简历 Vue 那条用"熟悉"非"精通",和状态基本匹配,不用改。被深挖原理可坦白"Vue 项目主要在 2020-2022,原理细节记得不深,能快速上手"。
