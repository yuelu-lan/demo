## 第 1 题 · 微前端方案对比与为什么选 qiankun

### 分析过程

这题是微前端的总纲,面试官想看你能不能横向对比几种方案,而不是只会 qiankun。简历写了微前端经验,这题必问。

**1. 先把方案分层**

主流微前端方案有四类,各自哲学不同:

- **iframe**:最古老、最隔离、最笨重
- **基于路由分发的微前端(qiankun / single-spa)**:主应用监听路由,动态加载子应用并挂载
- **Module Federation(模块联邦)**:Webpack 5 原生能力,运行时共享模块
- **自研**:组合路由分发 + 资源加载 + 沙箱

**2. 逐个剖析**

**iframe:**
- 优点:隔离最彻底(JS、CSS、路由全隔离),天然安全,接入零改造
- 缺点:UI 割裂(弹窗只能限在 iframe 内、白屏、滚动条)、通信难(postMessage)、性能差(每个 iframe 独立窗口)、SEO 不友好、URL 不同步
- 场景:对接完全不可信的第三方、遗留系统硬塞进来

**single-spa:**
- 优点:路由级微前端鼻祖,框架无关,只管加载生命周期,不强制沙箱
- 缺点:几乎不带沙箱和样式隔离,要自己搞;接入成本高,要改子应用入口导出 bootstrap/mount/unmount;没有预加载等开箱能力
- 定位:底层框架,qiankun 是它的封装增强

**qiankun:**
- 优点:基于 single-spa,额外白送 JS 沙箱(Proxy/快照)、CSS 隔离(shadowDOM/scoped)、预加载、全局状态、HTML entry 接入(子应用几乎零改造)、生产可用、中文生态
- 缺点:基于路由分发,子应用整体替换,粒度粗;Vite 子应用接入麻烦;Proxy 沙箱有兼容性坑(Safari 老版本)
- 场景:中后台多团队、多技术栈、按路由切分的应用

**Module Federation:**
- 优点:模块级共享,粒度细,可共享依赖(共享一个 React 实例),构建期声明、运行时加载,性能好
- 缺点:强绑定 Webpack 5,子应用要改构建配置;共享依赖版本协商复杂;跨技术栈能力弱;没有 qiankun 那种 HTML entry 的"零改造"体验
- 场景:同技术栈(都 Webpack 5)、需要细粒度模块共享、共享依赖减包

**3. 选型逻辑**

- 要彻底隔离 + 不可信第三方 → iframe
- 同技术栈 + 要共享模块/共享依赖 → Module Federation
- 多技术栈 + 路由切分 + 要开箱即用的沙箱/隔离/预加载 → qiankun
- 想完全掌控、团队有精力造轮子 → single-spa / 自研

**4. 为什么中后台常选 qiankun**

中后台典型特征:多团队、技术栈不统一(React/Vue 混)、按业务模块路由切分、子应用可独立部署、对隔离有要求但不需要 iframe 级别。qiankun 的 HTML entry + 开箱沙箱 + 预加载最匹配这个场景,接入成本最低。Module Federation 要求同构建体系,跨技术栈不友好;iframe 体验太差。

**5. 诚实边界**

qiankun 不是银弹:Vite 子应用接入痛苦、React18 createRoot 需配多实例、全局错误处理要自己补、保活原生不支持。选它是因为"接入成本 + 隔离能力 + 生态"的综合权衡,不是因为它完美。

### 参考答案

**四类方案对比:**

| 方案 | 隔离性 | 接入成本 | 粒度 | 适用场景 |
|------|--------|----------|------|----------|
| iframe | 最强(JS/CSS/路由全隔离) | 零改造 | 整页 | 不可信第三方、遗留系统 |
| single-spa | 弱(无内置沙箱) | 高(改子应用入口) | 路由级 | 想自研、要完全掌控 |
| qiankun | 中(JS沙箱+CSS隔离) | 低(HTML entry) | 路由级 | 多技术栈中后台 |
| Module Federation | 弱(要自己搞样式隔离) | 中(改构建配置) | 模块级 | 同构建体系、共享依赖 |

**为什么选 qiankun:** 多技术栈 + 路由切分 + 要开箱沙箱/隔离/预加载 + 接入成本最低(HTML entry 子应用几乎零改造),最匹配中后台场景。

**对比 Module Federation:** MF 强绑定 Webpack 5、跨技术栈弱、无零改造 entry;适合同构建体系细粒度共享模块。qiankun 粒度粗(整应用)但跨技术栈、接入成本低。

**一句话总结:** iframe 隔离最强但体验最差,single-spa 是底层无沙箱,qiankun 是 single-spa 的开箱增强版(沙箱+隔离+预加载),Module Federation 是模块级共享适合同栈。中后台多技术栈选 qiankun,同栈共享依赖选 MF。

## 第 2 题 · qiankun 的 JS 沙箱隔离机制

### 分析过程

这是 qiankun 最核心的原理题,考察"为什么需要沙箱、快照沙箱和 Proxy 沙箱的区别、多实例怎么办"。

**1. 为什么需要 JS 沙箱**

子应用如果不隔离,直接往 `window` 上挂东西(全局变量、全局函数、第三方库如 jQuery 的 `$`),多个子应用轮流加载时全局变量互相污染:子应用 A 挂了 `window.appConfig`,子应用 B 读到的是 A 的残留;子应用 A 卸载后它挂在 window 上的事件监听、定时器还在跑,造成内存泄漏。沙箱就是为了让每个子应用有自己的"伪 window",互不污染。

**2. qiankun 的两种沙箱**

qiankun 按是否支持多实例(同时挂载多个子应用)提供两种沙箱:

**快照沙箱(SnapshotSandbox,legacy 模式,不支持多实例):**
- 原理:子应用挂载前,遍历 `window` 拍快照;子应用运行时对 window 的修改直接作用于真 window;卸载时把 window 恢复到挂载前的快照(记录了哪些 key 被改/新增,逐个回滚)。
- 优点:兼容性好,不依赖 Proxy(老浏览器可用)。
- 缺点:**不支持多实例**(多个快照沙箱会互相打架,都操作真 window);遍历 window 性能差。
- 适用:`singular: true`(单实例模式,同一时刻只有一个子应用)。

**Proxy 沙箱(ProxySandbox,默认模式,支持多实例):**
- 原理:为每个子应用创建一个 fake window(空对象),用 `Proxy` 包裹。子应用访问/修改全局变量时,Proxy 拦截:读时先查 fake window,没有再 fallback 到真 window;写时只写到 fake window,不污染真 window。
- 优点:**支持多实例**(每个子应用有自己的 fake window,互不影响);不污染真 window,卸载直接丢 fake window;性能好(不遍历)。
- 缺点:依赖 Proxy,Safari 老版本(<10)不支持。
- 适用:`singular: false`(多实例模式)。

**3. 子应用代码怎么"跑进"沙箱**

qiankun 不是简单包个 Proxy 就行,子应用代码里 `window.xxx = 1` 这种是在全局执行,默认直接写到真 window。qiankun 的做法:用 `with` 语句把子应用代码包起来,with 的作用域对象指向 fake window 的 Proxy。这样代码里所有裸标识符(没显式写 `window.` 的全局变量访问)都会先查 Proxy 沙箱。配合子应用入口 JS 在沙箱环境执行,实现隔离。

**4. 沙箱逃逸的真实坑**

并非所有代码都能被沙箱罩住:
- **直接 `window.xxx =`**:Proxy 能拦住(取决于实现,新版 qiankun 拦截 set)。
- **通过 `globalThis`、`self`、`frames` 等别名访问全局**:可能绕过 Proxy,需要 qiankun 一并代理这些。
- **原生 DOM API 的全局监听**:`document.body.addEventListener`、`window.addEventListener('resize')` 这类,qiankun 要在卸载时兜底清理,否则监听残留。
- **第三方库直接读 `window` 引用**(如有些库启动时缓存了 `var g = window`):后续通过 `g.xxx` 改全局会逃逸,因为 g 指向真 window。

**5. `singular` 配置与沙箱选择**

`start({ singular: true/false })`:
- `singular: true`(旧版默认):同一时刻只激活一个子应用,用快照沙箱(兼容性好)。
- `singular: false`(React18 多实例、多个子应用同时显示):用 Proxy 沙箱。
- 新版 qiankun 推荐多实例 Proxy 沙箱,单实例也能用 Proxy。

**6. 面试加分点**

能讲出:"快照沙箱靠'挂载前拍照、卸载后回滚'实现隔离,不支持多实例;Proxy 沙箱靠'fake window + Proxy 拦截读写'实现隔离,支持多实例且不污染真 window。子应用代码靠 with 语句把作用域指向沙箱 Proxy。沙箱逃逸主要来自 globalThis 别名、原生事件监听、第三方库缓存 window 引用,这些是 qiankun 兜底处理的难点。"

### 参考答案

**为什么需要沙箱:** 子应用直接写 `window` 会导致全局变量污染(子应用 A 卸载后残留被 B 读到)和内存泄漏(事件监听/定时器残留)。沙箱给每个子应用一个"伪 window"。

**两种沙箱:**

| | 快照沙箱(SnapshotSandbox) | Proxy 沙箱(ProxySandbox) |
|---|---|---|
| 原理 | 挂载前拍照,卸载后回滚 window | fake window + Proxy 拦截读写 |
| 多实例 | 不支持 | 支持 |
| 是否污染真 window | 运行时污染,卸载回滚 | 不污染(只写 fake window) |
| 依赖 | 无 Proxy,兼容老浏览器 | 依赖 Proxy |
| 适用 | `singular: true` 单实例 | `singular: false` 多实例 |

**子应用如何跑进沙箱:** 用 `with` 语句把子应用代码包起来,作用域对象指向 fake window 的 Proxy,裸标识符访问全局时被 Proxy 拦截。

**沙箱逃逸的坑:** `globalThis`/`self`/`frames` 等别名绕过 Proxy、原生 DOM 全局监听(`addEventListener`)卸载需兜底清理、第三方库缓存 `var g = window` 后通过 `g` 改全局。

**singular 配置:** `singular: true` 单实例用快照沙箱;`singular: false` 多实例用 Proxy 沙箱。新版默认推荐 Proxy。

## 第 3 题 · qiankun 的 CSS 样式隔离

### 分析过程

JS 有沙箱,CSS 也得隔离,否则子应用的全局样式(`body { font-size }`、`.btn { color }`)会污染主应用和其他子应用。这题考察隔离手段和它们各自的坑。

**1. qiankun 提供的两种样式隔离**

通过 `start({ sandbox: { experimentalStyleIsolation: true } })` 或 `start({ sandbox: { strictStyleIsolation: true } })` 开启:

**strictStyleIsolation(严格隔离,shadowDOM):**
- 原理:把子应用挂到一个 Shadow DOM 节点里,Shadow DOM 天然样式隔离,子应用样式出不去、外部样式进不来。
- 致命问题:**第三方组件库(antd、Element)的 Modal/Drawer/Tooltip/Select 下拉等通过 `createPortal` 渲染到 `document.body`,不在 Shadow DOM 内,样式全部失效**——按钮变成无样式的裸 HTML。这是 shadowDOM 方案的死穴,生产几乎不能用。
- 场景:子应用极其简单、不依赖 portal 组件时可用,实际很少。

**experimentalStyleIsolation(实验性隔离,scoped css):**
- 原理:qiankun 在运行时给子应用所有 CSS 规则加一个父选择器前缀,形如 `div[data-qiankun="appName"] .btn { color }`,让子应用样式只在自己容器内生效。
- 优点:不破坏 portal 组件(因为还是挂在 body,只是样式被前缀限定到容器内,portal 组件挂 body 反而……这里仍有问题,见下)。
- 缺点:前缀方案对 `body`、`html` 这类全局选择器无效(无法变成 `div[data-qiankun] body`);动态插入的 style 标签要持续处理;性能有开销。
- 场景:多数项目的折中选择。

**2. 为什么 antd Modal 会逃逸**

这是面试高频追问点。antd 的 Modal/Drawer/Message 等组件默认通过 React Portal 渲染到 `document.body`(为了脱离父级 `overflow:hidden`、`z-index` 层叠上下文限制)。

- shadowDOM 模式:子应用在 Shadow DOM 内,portal 渲染到 body 的元素在 Shadow 外,子应用的 CSS 在 Shadow 内出不去 → Modal 无样式。
- scoped 模式:样式被前缀成 `div[data-qiankun] .ant-modal`,但 portal 挂在 body 上,不在 `div[data-qiankun]` 容器内 → 选择器匹配不到,同样无样式。

**3. 解法**

- **antd 配置 `getContainer`**:Modal/Drawer 等组件支持 `getContainer` prop,指定渲染到子应用容器内而非 body。但每个组件都要配,成本高,且可能引入层叠上下文问题。
- **全局 ConfigProvider**:antd5 的 ConfigProvider 可统一设 `getContainer` 或 popup container。
- **不用严格 shadowDOM**,用 scoped + 手动处理 portal 容器,或干脆子应用样式自己加命名空间(如 antd 的 prefixCls)。

**4. 实践中常见的折中**

很多团队不上 qiankun 的样式隔离,而是靠约定:
- 子应用用 CSS Modules / CSS-in-JS(样式天然 scoped 到组件)
- 全局样式加项目前缀(如 `.sub-app-xxx`)
- 组件库用 `prefixCls` / `ConfigProvider` 改类名前缀,避免 `.ant-btn` 这种全局类名冲突
- 主应用和子应用用不同 UI 库或不同版本时,prefixCls 错开

**5. 面试加分点**

能讲出:"qiankun 两种样式隔离:shadowDOM 严格但破坏 portal 组件(antd Modal 失效),scoped 前缀但管不了 body/html 选择器和 portal。生产中常靠 CSS Modules + 组件库 prefixCls + 命名约定手工隔离,而不是完全依赖 qiankun 内置方案。"

### 参考答案

**两种隔离:**

| | strictStyleIsolation | experimentalStyleIsolation |
|---|---|---|
| 原理 | Shadow DOM | 运行时给 CSS 规则加父选择器前缀 `div[data-qiankun=app] xxx` |
| 隔离强度 | 强(完全隔离) | 中(前缀限定) |
| 致命问题 | portal 组件(antd Modal/Drawer)渲染到 body,在 Shadow 外样式失效 | body/html 全局选择器无法加前缀;动态 style 要持续处理 |

**为什么 antd Modal 失效:** Modal 等通过 React Portal 渲染到 `document.body`,脱离子应用容器。shadowDOM 下子应用样式在 Shadow 内出不去;scoped 下前缀选择器匹配不到 body 上的 portal 元素。

**解法:**
- antd 组件配 `getContainer` 渲染到子应用容器内
- ConfigProvider 统一配置 popup 容器 / prefixCls
- 实践折中:不用内置隔离,靠 CSS Modules + 组件库 prefixCls + 命名约定手工隔离

**一句话总结:** qiankun 样式隔离有 shadowDOM(强但破坏 portal)和 scoped 前缀(中但管不了 body/portal)两种,生产常靠 CSS Modules + prefixCls 手工隔离。

## 第 4 题 · qiankun 应用间通信方案

### 分析过程

微前端应用间必然要通信:主应用传用户信息/权限给子应用、子应用通知主应用切换菜单、多子应用共享状态。这题考察通信方案选型。

**1. qiankun 内置:initGlobalState(Actions)**

qiankun 提供 `initGlobalState` 创建一个全局状态对象(内部基于发布订阅):

```js
// 主应用
import { initGlobalState } from 'qiankun';
const actions = initGlobalState({ user: null, token: '' });
actions.onGlobalStateChange((state, prev) => { /* 监听变化 */ });
actions.setGlobalState({ user: { name: 'xx' } });  // 触发所有子应用监听

// 子应用 mount 时收到 props,包含 onGlobalStateChange / setGlobalState
export function mount(props) {
  props.onGlobalStateChange((state) => { /* 收到主应用广播 */ });
  props.setGlobalState({ token: 'xxx' });  // 子应用也能广播
}
```

- 优点:官方内置,零依赖,主子双向通信。
- 缺点:状态是"广播"——所有子应用都收到,无法字段级订阅;数据不可追溯(没有 devtools);复杂状态管理弱;数据流不清晰(谁都能 set,易乱)。
- 适用:简单通信(传用户信息、token、主题)。

**2. 自定义事件 / EventEmitter**

主子应用约定一个全局事件总线(挂在 window 或通过 props 传):
- 主应用 `window.dispatchEvent(new CustomEvent('user-change', { detail }))`
- 子应用 `window.addEventListener('user-change', handler)`
- 优点:解耦、灵活。
- 缺点:事件名要约定,无类型,难追踪;沙箱下 window 事件要确认能穿透。

**3. Props 下发(主→子单向)**

主应用在 `registerMicroApps` 的生命周期或 `loadMicroApp` 时通过 props 传数据/回调给子应用:
```js
loadMicroApp({ name, entry, container, props: { user, onLogout } });
```
子应用 mount 收到 `props`。适合主→子的单向数据 + 子调主回调。最简单清晰,优先用。

**4. 共享状态库(子应用订阅外部 store)**

主应用维护一个状态库(redux/zustand),把 store 实例通过 props 传给子应用,子应用用 `useSyncExternalStore` 订阅。这样字段级订阅、devtools 可用、数据流清晰。适合复杂状态。这也是现代推荐的替代 initGlobalState 的方案。

**5. 通信方案选型**

- 主→子单向数据 + 子回调主 → **Props 下发**(最简单,优先)
- 简单全局广播(用户信息、主题、token) → **initGlobalState**
- 事件解耦(子应用间、跨层级通知) → **CustomEvent / EventEmitter**
- 复杂共享状态(字段级订阅、devtools) → **共享 store + useSyncExternalStore**

**6. 设计原则**

- **主子解耦**:子应用不应假设主应用一定传某数据,要能独立运行(便于本地开发)。
- **单向数据流优先**:props 下发 + 回调,比双向广播易维护。
- **通信最小化**:只传必要数据,别把整个状态树共享,否则子应用和主应用强耦合。

**7. 面试加分点**

能讲出:"qiankun 内置 initGlobalState 是广播式,无法字段级订阅、无 devtools,适合简单通信。复杂场景用 props 下发(单向,最清晰)或主应用 store + useSyncExternalStore(字段级订阅,可追溯)。通信设计要主子解耦,子应用能独立跑。"

### 参考答案

**qiankun 内置 initGlobalState:** 发布订阅式全局状态,主子双向广播。优点零依赖;缺点全量广播无字段级订阅、无 devtools、数据流易乱。适合简单通信(用户信息、token、主题)。

**四种方案选型:**

| 方案 | 方向 | 适用 |
|------|------|------|
| Props 下发(主→子 + 回调) | 单向 | 优先,最清晰 |
| initGlobalState | 双向广播 | 简单全局状态 |
| CustomEvent / EventEmitter | 事件解耦 | 子间通知、跨层级 |
| 共享 store + useSyncExternalStore | 双向字段级 | 复杂状态、要 devtools |

**设计原则:** 主子解耦(子应用能独立运行)、单向数据流优先、通信最小化(别共享整个状态树)。

**一句话总结:** 简单通信用 initGlobalState 或 props 下发,复杂状态用共享 store + useSyncExternalStore 字段级订阅。优先 props 单向下发,避免广播式通信导致数据流混乱。

## 第 5 题 · qiankun 路由隔离与 baseroute

### 分析过程

微前端路由是重灾区:主应用和子应用都有路由,base 路径怎么定、history/hash 模式冲突、activeRule 怎么匹配、子应用路由跳转会不会影响主应用。这题考察路由机制理解。

**1. 主子路由的关系**

qiankun 基于路由分发:主应用监听 URL,当 URL 匹配某子应用的 `activeRule` 时加载并挂载该子应用,把子应用挂到主应用的 `container` 节点。子应用内部再用自己的路由(react-router/vue-router)处理二级路径。

例如:URL `/sub-app/user/list` → 主应用匹配 `activeRule: '/sub-app'` → 加载子应用 → 子应用路由匹配 `/user/list` 渲染对应页面。

**2. baseroute:子应用的基础路径**

子应用路由不能假设自己在根路径 `/`,因为它实际运行在 `/sub-app` 下。子应用要把路由 base 设为 `/sub-app`。

- qiankun 在子应用 mount 时通过 props 传 `props.baseRoute`(主应用分配给它的基础路径)。
- 子应用 react-router 用 `<BrowserRouter basename={baseRoute}>`,vue-router 用 `base` 选项。
- 这样子应用内部 `<Link to="/user">` 生成的实际 URL 是 `/sub-app/user`,与主应用路由前缀对齐。

不设 baseroute 的后果:子应用跳转到 `/user`,实际 URL 变成 `/user`(丢了 `/sub-app` 前缀),主应用 activeRule 匹配不到 `/sub-app`,子应用被卸载,路由错乱。

**3. activeRule 的匹配**

`activeRule`(新版叫 `activeWhen`)决定何时激活子应用:
- 字符串:`'/sub-app'` —— URL 以该前缀开头时激活。
- 函数:`(location) => location.pathname.startsWith('/sub-app')` —— 自定义匹配逻辑,适合复杂场景(如多个路径激活同一子应用)。
- 数组:`['/sub-a', '/sub-b']` —— 多路径激活。

注意 activeRule 要互斥(不同子应用的 activeRule 别重叠),否则两个子应用同时激活(多实例)或冲突。

**4. history vs hash 模式冲突**

- 主应用 history 模式 + 子应用 history 模式:最常见,baseroute 对齐即可。
- 主应用 history + 子应用 hash:子应用用 hash 路由(`#/user`),URL 是 `/sub-app#/user`,hash 变化不触发主应用路由监听(history 模式监听 popstate 不监听 hashchange),子应用路由切换主应用无感知,可工作但 URL 不优雅。
- 主应用 hash + 子应用 history:容易冲突,主应用 hash 变化和子应用 history 变化互相干扰,不推荐。
- 推荐:主子都用 history 模式,base 对齐。

**5. urlRerouteOnly 配置**

`start({ urlRerouteOnly: true })`:qiankun 监听路由变化时,是否只在 URL 真正变化时才触发重新路由。默认 true,避免重复触发。这是性能优化项,理解即可。

**6. 子应用路由跳转影响主应用**

子应用用 `history.pushState` 跳转时,qiankun 会劫持路由变化,判断是否还在当前子应用 activeRule 范围内:是则子应用内部处理,否则触发主应用切换到别的子应用。这是 qiankun 路由分发的核心逻辑。

**7. 常见坑**

- baseroute 没设 → 子应用路由跳转丢前缀,子应用被卸载。
- activeRule 重叠 → 多子应用冲突。
- 主子路由模式不一致 → URL 错乱。
- 子应用用 `window.location.href = '/xxx'` 整页跳转 → 绕过 qiankun,导致整页刷新,丢失主应用。应该用 router 跳转。

**8. 面试加分点**

能讲出:"qiankun 基于路由分发,主应用按 activeRule 激活子应用,子应用通过 props.baseRoute 设路由 base 对齐前缀。主子都用 history 模式 + base 对齐最稳。activeRule 要互斥。子应用跳转要用 router 而非 location.href,否则整页刷新丢主应用。新版 qiankun 把 activeRule 改名为 activeWhen。"

### 参考答案

**机制:** 主应用监听 URL,匹配某子应用 `activeRule`(新版 `activeWhen`)时加载挂载该子应用,子应用内部用自己的路由处理二级路径。

**baseroute:** 子应用不能假设在根路径,mount 时收到 `props.baseRoute`,react-router 用 `<BrowserRouter basename={baseRoute}>` 对齐主应用分配的前缀。不设会导致子应用跳转丢前缀、被主应用卸载。

**activeRule 匹配:** 字符串前缀 / 函数 / 数组,需互斥避免冲突。

**路由模式:** 主子都用 history 模式 + base 对齐最稳;主 history + 子 hash 可工作但 URL 不优雅;主 hash + 子 history 易冲突不推荐。

**常见坑:**
- baseroute 没设 → 跳转丢前缀、子应用被卸载
- activeRule 重叠 → 多子应用冲突
- 子应用用 `window.location.href` 整页跳转 → 绕过 qiankun 整页刷新,应用 router 跳转

**一句话总结:** qiankun 路由分发靠 activeRule 激活子应用 + baseroute 对齐前缀,主子统一 history 模式,子应用跳转走 router 不走 location.href。

## 第 6 题 · qiankun 应用加载与 prefetch 预加载策略

### 分析过程

这题考察 qiankun 的加载流程和性能优化。`registerMicroApps` → `start` → 预加载的机制要清楚。

**1. 加载流程**

- `registerMicroApps(apps, lifeCycles)`:注册子应用配置(name、entry、container、activeRule),并挂全局生命周期钩子(beforeLoad/beforeMount/afterMount/afterUnmount)。此时不加载。
- `start(options)`:启动 qiankun,开始监听路由。当 URL 匹配某子应用 activeRule 时,触发加载:拉取子应用 entry(HTML)→ 解析出 JS/CSS 资源 → 在沙箱环境执行子应用入口 JS → 调用子应用导出的 `bootstrap` → `mount` 生命周期 → 挂载到 container。
- 子应用入口必须导出 `bootstrap`、`mount`、`unmount` 三个生命周期函数(qiankun 约定)。

**2. HTML entry 的意义**

qiankun 用 import-html-entry 拉取子应用的 HTML,解析其中的 `<script>`/`<link>` 标签,按顺序执行 JS、注入 CSS。好处:子应用不用改打包产物格式,正常构建出一个能独立运行的 HTML 即可,qiankun 自动解析资源。这就是"子应用几乎零改造"的来源(只需导出生命周期 + 配置允许跨域 + 改 publicPath)。

**3. prefetch 预加载**

`start({ prefetch: true })` 控制是否在空闲时预加载未激活的子应用资源,提升切换速度:

- `prefetch: true`(默认):在第一个子应用 mount 完成后,用 `requestIdleCallback` 在浏览器空闲时预加载其他子应用的 JS/CSS(只下载不执行)。
- `prefetch: false`:关闭预加载,切到才加载(首次切换慢)。
- `prefetch: 'all'`:主应用 start 后立即预加载所有子应用。
- `prefetch: (apps) => apps.filter(...)`:自定义函数,返回要预加载的子应用列表,精细控制(如只预加载高频子应用)。

**4. 预加载的本质**

预加载只下载资源(JS/CSS 文本),不执行、不挂载。等真正激活该子应用时,资源已在缓存,直接执行,省掉网络往返。适合子应用资源大、切换频繁的中后台。

**5. 何时关 prefetch**

- 子应用很多 + 资源很大:全预加载会占带宽和内存,首屏变慢,可关或用自定义函数只预加载头几个。
- 弱网移动端:预加载可能挤占首屏关键资源。
- 开发环境:关掉避免干扰。

**6. loadMicroApp 手动加载**

除了 `registerMicroApps` + 路由自动激活,qiankun 还提供 `loadMicroApp(app, config, lifeCycles)` 手动加载子应用到指定容器。适合:一个页面里同时显示多个子应用(多实例)、非路由驱动的动态加载(如点击按钮在某区域加载子应用)。返回 MicroApp 实例,可手动 unmount。

**7. singular 与加载的关系**

- `singular: true`(单实例):同一时刻只一个子应用,切到新的自动卸载旧的。
- `singular: false`(多实例):多个子应用可同时挂载,需用 `loadMicroApp` 手动管理或多个 activeRule 同时命中。

**8. 面试加分点**

能讲出:"registerMicroApps 只注册不加载,start 后监听路由按需加载,通过 HTML entry 拉取子应用 HTML 解析资源实现零改造接入。prefetch 默认在首个子应用 mount 后用 requestIdleCallback 空闲预加载其他子应用资源(只下载不执行),可配 true/false/'all'/自定义函数。多实例或动态加载用 loadMicroApp 手动控制。"

### 参考答案

**加载流程:** `registerMicroApps` 注册(不加载)→ `start` 监听路由 → URL 匹配 activeRule 时拉取子应用 entry(HTML)→ 解析 JS/CSS → 沙箱执行 → 调用子应用 `bootstrap`/`mount` → 挂载到 container。子应用须导出 bootstrap/mount/unmount 生命周期。

**HTML entry:** qiankun 用 import-html-entry 拉取子应用 HTML 并解析 `<script>`/`<link>`,子应用正常构建无需改产物格式,这是"零改造接入"的来源。

**prefetch 预加载:**

| 值 | 行为 |
|----|------|
| `true`(默认) | 首个子应用 mount 后,requestIdleCallback 空闲预加载其他子应用资源(只下载不执行) |
| `false` | 关闭,切到才加载 |
| `'all'` | start 后立即预加载所有 |
| 函数 | 自定义返回要预加载的子应用列表 |

**何时关 prefetch:** 子应用多且资源大(占带宽内存)、弱网、开发环境。或用自定义函数只预加载高频子应用。

**loadMicroApp:** 手动加载子应用到指定容器,适合多实例同显或动态非路由加载,返回实例可手动 unmount。

**singular:** `true` 单实例(切新自动卸载旧),`false` 多实例(同时挂载多个)。

**一句话总结:** registerMicroApps 注册 → start 路由驱动加载 → HTML entry 零改造接入 → prefetch 空闲预加载提速度 → loadMicroApp 手动多实例。

## 第 7 题 · qiankun 子应用保活(keep-alive)

### 分析过程

qiankun 默认切换子应用时卸载(unmount)旧的、挂载新的,旧子应用状态丢失(表单未保存数据、滚动位置、组件内部 state)。用户期望像 vue keep-alive 那样"切走不销毁、切回恢复"。这题考察保活方案。

**1. qiankun 默认为什么卸载**

qiankun 基于 single-spa,切换子应用调用 `unmount` 生命周期,子应用从 container 移除。这是路由分发微前端的固有行为——container 同一时刻只承载一个子应用。卸载后子应用 JS 上下文还在(没刷新页面),但 DOM 和 React/Vue 组件树被销毁,state 丢失。

**2. qiankun 原生不支持 keep-alive**

qiankun 没有内置 keep-alive,因为:
- unmount 是 single-spa 协议约定,子应用被卸载时理应清理。
- 保活意味着 DOM 不销毁,但 container 要腾给新子应用,DOM 放哪是个问题。
- 保活子应用的后台行为(定时器、监听、网络请求)仍在跑,资源占用。

**3. 实现保活的思路**

**思路 A:display:none 隐藏而非卸载(loadMicroApp + 手动隐藏容器)**

用 `loadMicroApp` 手动加载多个子应用,每个子应用有自己的 container。切换时不是 unmount,而是把旧子应用容器 `display:none`,新子应用容器 `display:block`。子应用 DOM 和 state 保留。

- 优点:真正保活,state/滚动位置/表单全保留。
- 缺点:多个子应用 DOM 同时存在(只是隐藏),内存占用大;隐藏的子应用仍在跑定时器/监听,要自己管理;要手动管理容器显示切换,失去 qiankun 路由自动分发能力。
- 适用:子应用数量不多、确实需要保活(如表单填写中断切走再切回)。

**思路 B:子应用自身实现 keep-alive**

子应用内部用 react-router 的 keep-alive 库(如 react-activation)或 vue 的 `<keep-alive>` 缓存内部路由组件。qiankun 层面仍卸载子应用,但子应用再次 mount 时,内部 keep-alive 恢复之前的组件 state。

- 优点:不破坏 qiankun 路由分发,子应用自治。
- 缺点:qiankun unmount 时整个子应用组件树销毁,纯组件级 keep-alive 救不了"子应用被卸载"这件事——除非子应用的 mount 能检测到"我之前挂过"并恢复。实际效果有限。
- 适用:子应用内部页面间保活,非子应用间保活。

**思路 C:不卸载,用 v-show/display 控制显隐的微前端框架**

有些方案(如 wujie)原生支持 keep-alive,因为它们不是 unmount 模式而是 iframe/隐藏容器模式。如果保活是硬需求,可考虑 wujie 而非 qiankun。这是选型时的考量。

**4. qiankun 保活的真实做法**

生产中 qiankun 保活常用思路 A 的变体:
- 主应用为每个需要保活的子应用建独立 container。
- 用 `loadMicroApp` 加载,切换时 toggle container 的 `display`。
- 配合 `singular: false` 多实例。
- 自己处理路由与 container 显隐的映射(不再用 registerMicroApps 自动分发)。
- 隐藏子应用的副作用(定时器/监听)由子应用在 `unmount` 时暂停、`mount` 时恢复(但保活不调 unmount,所以要子应用暴露 pause/resume 生命周期或监听可见性)。

**5. 保活的代价**

- 内存:多个子应用 DOM + JS 上下文常驻。
- 性能:隐藏子应用的后台行为仍耗资源。
- 复杂度:失去自动路由分发,要手动管理显隐和副作用。
- 滚动/表单恢复:需额外处理(隐藏时记滚动位置,显示时恢复)。

**6. 何时需要保活**

- 表单填写中途切走(怕丢数据)。
- 列表滚动位置要保留。
- 子应用初始化慢,不想每次重新加载。

但多数场景:表单数据应存到 store/localStorage(而非靠保活),列表位置可记路由 query,初始化慢靠 prefetch 解决。保活是最后手段,不是首选。

**7. 面试加分点**

能讲出:"qiankun 原生不支持 keep-alive,因为 single-spa 协议的 unmount 会销毁子应用。实现保活用 loadMicroApp 多实例 + 每个 container display:none 切换显示,代价是内存占用大、要手动管理显隐和副作用、失去自动路由分发。保活是重手段,优先考虑 store 持久化表单、路由 query 记状态、prefetch 提速等轻方案。如果保活是硬需求,可考虑 wujie 等原生保活框架。"

### 参考答案

**为什么需要保活:** 默认切换子应用调用 unmount 销毁 DOM 和组件树,表单数据/滚动位置/internal state 丢失。

**qiankun 原生不支持:** single-spa 协议 unmount 约定销毁,container 同时刻只载一个子应用,DOM 放哪 + 后台副作用管理是难题。

**实现思路:**
1. **loadMicroApp + container display 切换**(主用):多实例每个子应用独立 container,切换 toggle display 而非 unmount,真正保活。代价:内存大、副作用手动管、失去自动路由分发。
2. **子应用内部 keep-alive**(react-activation/vue keep-alive):子应用内页面间保活,但救不了子应用整体卸载。
3. **换框架**:wujie 等原生支持保活,保活是硬需求时的选型考量。

**代价:** 内存占用(多 DOM 常驻)、后台副作用仍耗资源、手动管理显隐和副作用、失去自动路由分发。

**优先轻方案:** 表单存 store/localStorage、列表位置记路由 query、初始化慢用 prefetch。保活是最后手段。

**一句话总结:** qiankun 原生无 keep-alive,用 loadMicroApp 多实例 + container display 切换实现,代价大;优先用 store 持久化 + prefetch 等轻方案,硬需求可考虑 wujie。

## 第 8 题 · qiankun 的局限与常见踩坑

### 分析过程

这题考察"你是不是真用过 qiankun 还是只看过文档",踩坑细节决定可信度。简历写微前端经验,这题是验真题。

**1. Vite 子应用接入难**

qiankun 的沙箱用 `with` + eval 执行子应用 JS,要求子应用入口 JS 是可被 with 包裹执行的格式(传统 UMD/CJS)。Vite 默认产物是 ESM(`<script type="module">`),ESM 不能被 with 包裹,且 import 是静态的,沙箱拦不住。

解法:
- Vite 子应用用 `vite-plugin-qiankun` 插件,把入口改造成 UMD 兼容格式。
- 或子应用用 legacy 模式构建出非 ESM 产物。
- 或主应用不用沙箱(`sandbox: false`,但失去隔离)。
这是 qiankun + Vite 的老大难,面试高频。

**2. React 18 createRoot 兼容**

React 18 的 `createRoot` 和 `ReactDOM.render` 行为不同,qiankun 多次 mount/unmount 同一子应用时,React 18 的 root 复用和卸载逻辑与 qiankun 生命周期配合有坑:
- 子应用要在 mount 时 `createRoot` 并 render,unmount 时 `root.unmount()` 彻底清理。
- 不能缓存 root 实例跨 mount(unmount 后 root 失效)。
- 多实例(`singular: false`)下多个 root 要分开管理。
- React18 StrictMode 双调用与 qiankun 生命周期叠加可能重复 mount。

解法:子应用每次 mount 新建 root,unmount 调 root.unmount(),不要复用。

**3. Safari Proxy 兼容**

Proxy 沙箱依赖 ES6 Proxy,Safari < 10 不支持。老 iOS/老 Safari 上 Proxy 沙箱失效,降级到快照沙箱(但快照不支持多实例)。现在主流 Safari 都支持,但兼容老设备时要注意。

**4. 全局错误处理**

子应用加载失败(资源 404、JS 执行报错、生命周期导出缺失)默认可能静默,qiankun 提供 `addGlobalUncaughtErrorHandler` 捕获。生产要注册错误处理,否则子应用挂了用户只看到白屏无提示。

**5. 资源跨域(CORS)**

子应用 HTML/JS/CSS 由主应用 fetch 拉取,子应用服务器必须配 CORS 允许主应用域访问,否则拉不到资源。开发环境子应用 dev server 也要配 `Access-Control-Allow-Origin`。生产同域部署可规避,跨域部署必配 CORS。

**6. publicPath 动态配置**

子应用资源路径基于自身 publicPath,但被主应用加载后,子应用的相对路径资源(图片、异步 chunk)可能解析到主应用域下导致 404。子应用要用 `__webpack_public_path__` 动态设置(运行时根据 qiankun 注入的路径修正),或用绝对路径部署。qiankun 文档有 `public-path.js` 示例。

**7. 全局事件 / 定时器泄漏**

子应用 addEventListener、setInterval 在 unmount 时若不清理,卸载后仍跑,造成内存泄漏和报错(操作已卸载 DOM)。子应用 unmount 生命周期必须清理所有全局监听和定时器。qiankun 沙箱会兜底清理部分,但不能完全依赖。

**8. 样式冲突(见第 3 题)**

antd Modal portal 逃逸、全局样式污染,prefixCls + CSS Modules 手工隔离。

**9. 嵌套子应用 / 主子应用同框架版本冲突**

主子都用 React 但版本不同(如主 React17 子 React18),两个 React 实例可能冲突(尤其全局事件系统)。解法:子应用独立打包 React(不共享),或用 Module Federation 共享指定版本。

**10. SEO / 首屏**

qiankun 是客户端动态加载,首屏主应用先空壳再加载子应用,SEO 不友好、首屏有白屏。中后台无所谓,ToC 慎用。可配 prefetch 缓解。

**11. 面试加分点**

能讲出 3-5 个具体坑 + 解法,证明真用过。重点:Vite ESM 沙箱不兼容(vite-plugin-qiankun)、React18 root 生命周期、CORS、publicPath 动态修正、全局事件泄漏清理、addGlobalUncaughtErrorHandler 错误兜底。

### 参考答案

**主要局限与踩坑:**

| 坑 | 原因 | 解法 |
|----|------|------|
| Vite 子应用接入难 | ESM 不能被 with 包裹,沙箱拦不住 | `vite-plugin-qiankun` 改 UMD,或 legacy 构建 |
| React18 createRoot | root 复用/卸载与生命周期冲突 | 每次 mount 新建 root,unmount 调 `root.unmount()` |
| Safari Proxy | Proxy 沙箱依赖 ES6,Safari<10 不支持 | 降级快照沙箱(但失去多实例) |
| 资源跨域 | 主应用 fetch 子应用资源 | 子应用配 CORS,dev server 也要配 |
| publicPath 错误 | 相对路径资源解析到主应用域 | `__webpack_public_path__` 动态修正或绝对路径 |
| 全局事件/定时器泄漏 | unmount 未清理 | unmount 生命周期清理所有监听和定时器 |
| 错误静默 | 加载失败无提示 | `addGlobalUncaughtErrorHandler` 兜底 |
| 样式冲突 | portal 逃逸、全局样式 | prefixCls + CSS Modules 手工隔离(见第3题) |
| SEO/首屏白屏 | 客户端动态加载 | prefetch 缓解,ToC 慎用 |

**高频验真坑:** Vite ESM 沙箱不兼容、React18 root 生命周期、CORS、publicPath 动态修正、全局事件泄漏清理、addGlobalUncaughtErrorHandler 错误兜底。能说出 3-5 个 + 解法即证明真用过。

**一句话总结:** qiankun 主要坑在 Vite ESM 沙箱不兼容、React18 root 生命周期、CORS、publicPath、全局副作用泄漏、错误静默;生产要逐项配齐解法,ToC 慎用(SEO/首屏)。
