## 第 1 题 · React 函数组件与类组件分别有什么优势

### 分析过程

面试官表面是对比两种组件写法,实际考察三点:是否理解 React 的设计演进、能否说出真实适用场景、而不是背"Hooks 万能"的教条。

拆解维度:

**1. 心智模型层面**
- 类组件基于 OOP:通过 `this`、`this.state`、生命周期方法组织逻辑,状态和副作用绑定在实例上
- 函数组件基于 FP:组件本身就是 `props => UI` 的纯函数,状态和副作用通过 Hooks "挂载"进来,没有实例概念

**2. 状态与逻辑复用**
- 类组件靠 HOC(高阶组件)和 render props 复用逻辑,容易出现"嵌套地狱",且 `this` 指向问题、`bind` 绑定是常见坑
- 函数组件用自定义 Hooks 复用,逻辑可扁平组合,没有组件层级嵌套

**3. 生命周期 vs Hooks**
- 类组件的生命周期分散了同一业务逻辑(比如订阅要在 `componentDidMount` 开、`componentWillUnmount` 关),逻辑被时间切分
- Hooks 的 `useEffect` 把相关逻辑聚合在一起,按"功能"而非"时间"组织

**4. 性能与心智负担**
- 类组件有 `shouldComponentUpdate`、`PureComponent`,但容易因为引用相等判断失误导致无谓渲染
- 函数组件配合 `React.memo`、`useMemo`、`useCallback` 更细粒度,但闭包陷阱(stale closure)是新坑

**5. 类组件仍未被完全废弃的优势**
- 错误边界(Error Boundaries)目前仍必须用类组件实现,因为 `getDerivedStateFromError` / `componentDidCatch` 还没有 Hooks 等价物
- 某些依赖实例引用的库(如某些旧版 `refs`、`DnD` 场景)类组件更直观

**6. 趋势**
React 官方文档已将函数组件 + Hooks 作为推荐写法,类组件进入"维护模式"而非"推荐模式",但不会强制移除。

### 参考答案

**函数组件优势:**
- 写法更简洁,没有 `this`、`bind`、构造函数等心智负担
- 逻辑复用通过自定义 Hooks 实现,避免 HOC / render props 的嵌套地狱
- 副作用按"功能"聚合在 `useEffect` 中,而非按生命周期时间点分散
- 更易做静态分析与 Tree Shaking,官方推荐写法

**类组件优势:**
- 生命周期划分清晰,对从命令式编程转过来的开发者更直观
- 实例上能稳定持有引用(`this`),适合依赖实例的旧库集成
- **错误边界目前仍只能用类组件实现**(`getDerivedStateFromError` / `componentDidCatch`)
- 已有大型存量代码库稳定性高,迁移成本低时无需强行重写

**一句话总结:** 函数组件是 React 的未来方向,逻辑复用和组合性更好;类组件在错误边界和存量维护场景仍有不可替代价值。新项目首选函数组件,老项目按需渐进迁移。

## 第 2 题 · 受控组件与非受控组件的优缺点

### 分析过程

核心是"谁来持有表单数据源"这个判断。

**1. 概念边界要清晰**
- 受控组件:表单元素的值由 React state 驱动,`value` 绑定 state,`onChange` 回写 state。React 是"唯一数据源"
- 非受控组件:表单元素自己持有 DOM 内部状态,React 通过 `ref` 在需要时读取,`defaultValue` 仅设初值。DOM 是数据源

**2. 从使用场景倒推优缺点**

受控组件适合:
- 需要实时校验(如输入时提示格式错误)
- 需要根据输入动态控制 UI(如禁用按钮、联动下拉)
- 数据需要立即提交或持久化

非受控组件适合:
- 一次性提交场景(注册表单最后点提交才校验)
- 集成非 React 代码(第三方 jQuery 插件)
- 简单场景下减少 re-render

**3. 性能维度**
- 受控组件每次按键都触发 setState → re-render,大表单或低性能场景有压力
- 非受控组件不触发 React 重渲染,输入流畅度高

**4. 心智负担**
- 受控组件逻辑集中、可预测、易测试,但样板代码多
- 非受控组件代码少,但"数据藏在 DOM 里",调试和同步状态时要小心 stale 问题

**5. React 官方立场**
官方文档明确建议:**大多数情况下用受控组件**,非受控是特定场景的优化手段,不是默认选择。

### 参考答案

**受控组件优点:**
- 数据流单向、可预测,状态集中在 React,便于校验、联动、持久化
- 实时控制 UI(禁用按钮、格式化输入、错误提示)
- 易于测试和回溯

**受控组件缺点:**
- 每次输入触发 re-render,大表单有性能压力
- 样板代码多,每个字段都要绑定 state 和 onChange

**非受控组件优点:**
- 不触发 re-render,输入流畅
- 代码量少,集成非 React 库更方便
- 适合一次性提交场景

**非受控组件缺点:**
- 数据藏在 DOM,状态不可预测,难以实时校验和联动
- 需要 `ref` 读取,逻辑分散

**选型原则:** 默认用受控组件;仅在一次性提交、集成第三方、或性能瓶颈时考虑非受控。复杂表单可考虑 `react-hook-form` 这类库,本质上是非受控 + 按需受控的折中。

## 第 2.5 题 · "非受控 + 按需受控"是什么

### 分析过程

这是从第 2 题延伸出来的表单库设计哲学问题,不是只会背"受控 vs 非受控"。

**1. 先回到两种纯模式的痛点**

- 纯受控:每个字段 setState → 每次 re-render,10 个字段的大表单每按一个键整个表单组件重渲染,性能差
- 纯非受控:DOM 持有值,React 想读要靠 ref,想实时校验、联动、控制按钮 disabled 就很难做

两者各有死穴:受控死在性能,非受控死在可控制性。

**2. "非受控 + 按需受控"的核心思想**

把"数据存储"和"数据驱动 UI"解耦:
- **底层用非受控**:表单值存在一个外部容器里(DOM 节点的 ref,或一个外部 store,如 `react-hook-form` 用的是非响应式的 ref 对象),不在 React state 里。这样输入时**不触发 React re-render**
- **上层按需受控**:当某个字段真的需要驱动 UI 时(比如根据输入实时禁用按钮、显示错误),再把那一个字段"提升"为受控,单独用 state 订阅

本质是:**默认走非受控的低成本路径,只在需要响应式的地方付受控的代价。**

**3. 以 react-hook-form 为例说明**

- `register` 把 input 注册到内部的 ref store,输入直接写 DOM,React 不参与 → 非受控
- `watch("field")` 或 `useWatch` 把某个字段变成订阅式受控,组件在该字段变化时 re-render → 按需受控
- `Controller` 组件则是给"必须受控"的第三方组件(如自定义 Select)用的受控适配器

这样就做到了:90% 字段零 re-render,10% 需要联动的字段按需响应。

**4. 为什么这种模式性能好**

关键在于 React 的 re-render 是按组件粒度的。纯受控把整个表单组件拖下水;按需受控只让订阅了该字段的组件 re-render。配合 `useWatch` 还能做组件级别的隔离订阅。

**5. 心智模型一句话**

> "数据住在 ref 里(非受控),谁需要听它的变化就用订阅把它变成局部的受控。"

### 参考答案

**核心思想:** 数据默认存在 ref / 外部 store(非受控,不触发 re-render),只有当某个字段需要驱动 UI 时,再通过订阅把它"提升"为局部的受控状态。

**以 react-hook-form 为例:**
- `register(name)` → 注册到内部 ref store,输入直接写 DOM,零 re-render(非受控)
- `watch(name)` / `useWatch` → 订阅某字段变化,组件按需 re-render(按需受控)
- `Controller` → 给必须受控的第三方组件做适配

**优点:**
- 大表单性能好,默认不触发 re-render
- 又保留了实时校验、联动、动态禁用等受控能力
- 校验、提交、错误状态都由库统一管理

**适用场景:** 字段多、有联动校验需求、对输入流畅度敏感的复杂表单。简单表单用原生受控就够了,不必上库。

**一句话总结:** 用非受控省掉默认的渲染成本,用按需受控买回需要的响应式能力——这是现代表单库的本质设计。

## 第 3 题 · ahooks 的 useRequest 相比 useEffect 发请求有哪些优势

### 分析过程

考察对请求场景的工程化抽象能力。面试官不是想听 API 对比,而是看你能不能识别"裸写 useEffect 发请求"会踩哪些坑,然后看 ahooks 是怎么系统性解决这些坑的。

**1. 先复盘裸 useEffect 发请求的典型写法**

```js
useEffect(() => {
  fetch('/api/user')
    .then(res => res.json())
    .then(setData);
}, [id]);
```

看起来没问题,但实际场景一展开全是坑。

**2. 逐个识别痛点**

- **竞态问题(race condition)**:id 从 1 变到 2,第一个请求慢于第二个,先返回的 2 被后返回的 1 覆盖,UI 显示错误数据。裸 useEffect 不会自动取消
- **内存泄漏 / setState on unmounted**:组件卸载后请求才返回,调用 setState 触发警告(React 18 已移除警告,但仍是隐患)
- **手动清理样板代码**:要用 AbortController 或布尔 flag,每次都写一遍
- **没有 loading / error / data 状态管理**:要自己维护三个 state,每个请求写一遍
- **无缓存**:来回切换 id 都重新请求,没有请求结果复用
- **无防抖/节流/轮询/依赖突变轮询**:搜索框、实时数据这些场景要自己造轮子
- **无聚焦重新请求、无屏幕重连重试**:用户体验场景缺失
- **错误重试机制缺失**:失败就失败,要自己写 retry
- **并发请求 / 并行依赖管理缺失**:多个请求组合要自己 Promise.all
- **TypeScript 类型推导弱**:自己写的没有统一类型约束

**3. 对比 useRequest 的系统性解法**

ahooks 的 `useRequest` 本质是把"请求生命周期 + 副作用控制 + 缓存策略 + 交互策略"打包成声明式 API:

- 自动管理 `loading / data / error`
- 自动处理竞态(内置请求取消)
- 自动处理卸载(unmount 丢弃响应)
- `cacheKey` 实现请求结果缓存与 SWR
- `debounceWait` / `throttleWait` 内置防抖节流
- `pollingInterval` / `pollingWhenHidden` 轮询
- `refreshOnWindowFocus` 聚焦重新请求
- `retryCount` / `retryInterval` 错误重试
- `refreshDeps` 替代手写依赖数组
- `runAsync` 返回 Promise,便于链式处理
- 完善的 TS 类型推导

**4. 抽象层次的升华**

裸 useEffect 是"机制层",useRequest 是"策略层"。useEffect 给你最大自由但也最大负担;useRequest 把高频请求模式沉淀为配置项,让业务代码只关心"请求什么",不关心"怎么安全地请求"。

**5. 边界要诚实**

useRequest 不是银弹:一次性、极简、无竞态风险的请求用裸 useEffect 也 OK;但只要涉及缓存、轮询、防抖、重试、竞态任一项,useRequest 都更稳。

### 参考答案

**裸 useEffect 发请求的痛点:**
- **竞态**:依赖变化时旧请求覆盖新数据,需手动 AbortController
- **卸载隐患**:组件卸载后 setState,需手动 flag 清理
- **状态样板**:`loading / data / error` 三件套每次手写
- **无缓存**:重复请求无复用
- **交互策略缺失**:防抖、节流、轮询、聚焦重请求、错误重试全要自己造

**useRequest 的系统性优势:**

| 能力 | useEffect | useRequest |
|------|-----------|------------|
| 竞态处理 | 手动 | 自动取消旧请求 |
| 卸载安全 | 手动 flag | 自动丢弃响应 |
| 状态管理 | 手写三件套 | `data/loading/error` 内置 |
| 缓存 / SWR | 无 | `cacheKey` 内置 |
| 防抖节流 | 手动 | `debounceWait` / `throttleWait` |
| 轮询 | 手动 | `pollingInterval` |
| 聚焦重请求 | 手动 | `refreshOnWindowFocus` |
| 错误重试 | 手动 | `retryCount` |
| 依赖触发 | 依赖数组 | `refreshDeps` |
| TS 类型 | 弱 | 完整推导 |

**本质区别:** useEffect 是"机制层",给自由但负担全在自己;useRequest 是"策略层",把高频请求模式(竞态、缓存、轮询、重试)沉淀为声明式配置,业务只关心"请求什么"。

**一句话总结:** 一次性简单请求用 useEffect 足够;但凡涉及竞态、缓存、轮询、防抖、重试任一项,useRequest 都更稳、更少 bug、更少样板代码。

## 第 4 题 · Claude Code、Codex、Cursor、GitHub Copilot 各自有什么优缺点

### 分析过程

考察对 AI 编程工具生态的认知广度。面试官想看你是不是真的用过、能不能客观对比,而不是站队某一家。

**1. 先分类定位,避免乱比**

这四个工具其实不在同一个赛道,要按"产品形态"先分层:

- **GitHub Copilot**:IDE 插件形态,嵌入式补全 + 对话,主打"隐形融入现有工作流"
- **Cursor**:VS Code fork 的独立 IDE 形态,主打"深度改造编辑器体验,Agent + 上下文管理"
- **Claude Code**:CLI / 终端原生 Agent 形态,主打"文件系统级自主操作、长任务自治"
- **Codex**:OpenAI 的 CLI/云端 Agent 形态(Codex CLI + Codex 云端任务),对标 Claude Code,主打"OpenAI 模型生态 + 异步云端执行"

**2. 逐个剖析优缺点**

**Copilot:**
- 优点:生态最广、几乎全 IDE 支持、企业渗透率高、补全体验成熟稳定、和 GitHub 生态深度绑定(PR、issue 联动)
- 缺点:Agent 能力相对弱、多文件大改不如独立 Agent、上下文理解不如 Claude/Cursor 深、模型选择受限

**Cursor:**
- 优点:编辑器内 Agent 体验打磨最好、Tab 补全 + Composer + Agent 模式衔接顺滑、上下文管理(`@file`、`@docs`、`@web`)直观、可切多模型
- 缺点:要换 IDE(从 VS Code 迁移成本低但仍是 fork)、订阅价格、对终端长任务自治不如 CLI Agent、企业合规门槛

**Claude Code:**
- 优点:终端原生、文件系统级自主操作、长任务自治能力强、子 Agent / 工作流编排、MCP 生态、可跑 CI/服务器/无头场景、Claude 模型推理强
- 缺点:CLI 上手门槛高于 GUI、无图形编辑器预览、token 消耗可能大、对纯前端可视化调试不如 Cursor 直观

**Codex:**
- 优点:OpenAI 模型生态(GPT 系列)、云端异步任务能力(开任务后台跑)、和 ChatGPT/OpenAI 账号体系打通、CLI 形态可融入终端工作流
- 缺点:生态成熟度不如 Copilot、编辑器集成不如 Cursor、Agent 编排能力与 MCP 生态不如 Claude Code 丰富、模型在长程代码推理上评价波动

**3. 抽象出对比维度**

可以从几个轴横向比:
- 形态:IDE 插件 vs 独立 IDE vs CLI Agent vs 云端 Agent
- 自治程度:补全 < 对话 < 单文件 Agent < 多文件 Agent < 终端自治 Agent
- 上下文深度:Copilot < Cursor ≈ Codex < Claude Code(长任务)
- 适用人群:个人补全 / 重度 IDE 用户 / 终端原生开发者 / 云端异步场景

**4. 诚实表态**

没有银弹,选型看场景:日常补全 Copilot,重 IDE 深度开发 Cursor,长任务/服务器/CI 场景 Claude Code,偏好 OpenAI 生态或云端异步选 Codex。很多团队会组合使用。

**5. 趋势观察**

工具正在从"补全"走向"Agent 自治",从"单点工具"走向"工作流编排"。MCP 协议的兴起让工具间边界变模糊。

### 参考答案

**GitHub Copilot**
- 优点:全 IDE 覆盖、生态最广、企业渗透率高、补全成熟、与 GitHub PR/Issue 联动
- 缺点:Agent 自治能力偏弱、多文件大改不如独立 Agent、模型选择受限、上下文深度一般

**Cursor**
- 优点:编辑器内 Agent 体验打磨最好、Tab + Composer + Agent 衔接顺滑、`@file/@docs/@web` 上下文管理直观、可切换多模型
- 缺点:需迁移到 fork 版 IDE、订阅成本、终端长任务自治不如 CLI Agent、企业合规门槛

**Claude Code**
- 优点:终端原生、文件系统级自主操作、长任务自治强、子 Agent / 工作流编排、MCP 生态丰富、可跑 CI/无头/服务器场景、Claude 模型推理强
- 缺点:CLI 上手门槛高、无图形化编辑预览、token 消耗可能大、前端可视化调试不如 Cursor 直观

**Codex**
- 优点:OpenAI 模型生态、云端异步任务(后台开任务跑)、与 ChatGPT 账号体系打通、CLI 可融入终端
- 缺点:生态成熟度不及 Copilot、编辑器集成不如 Cursor、Agent 编排与 MCP 生态不及 Claude Code 丰富、长程代码推理评价波动

**选型逻辑(按场景):**
- 日常补全、全 IDE → Copilot
- 重度 IDE 深度开发、可视化交互 → Cursor
- 终端原生、长任务自治、CI/服务器、工作流编排 → Claude Code
- 偏好 OpenAI 生态、云端异步任务 → Codex

**一句话总结:** 形态决定定位——Copilot 是"补全插件",Cursor 是"Agent IDE",Claude Code 是"终端自治 Agent",Codex 是"OpenAI 生态的云端/CLI Agent"。无银弹,按场景组合使用是常态。
